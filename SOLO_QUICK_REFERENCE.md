# Solo Quick Reference Card

> **⚠️ Platform Requirements:** Solo can only run on **macOS** or **Linux**. Windows users must use WSL2.
>
> **RAM Requirements:**
>
> -   Single node: Minimum **12 GB RAM**
> -   Dual node: Minimum **24 GB RAM** (required for dynamic address book tests)
>
> 📚 [Official Solo Documentation](https://solo.hiero.org/)

## Essential Commands

```bash
# ONE-TIME: Initialize Solo infrastructure (cluster, config, deployment)
task solo:init                           # Single node (12 GB RAM)
task solo:init -- --num-nodes 2          # Dual node (24 GB RAM, for DAB tests)

# START: Deploy and start services
task solo:start                          # Start all services

# STOP: Stop services (keeps cluster for fast restart)
task solo:stop                           # Stops services, preserves infrastructure

# TEARDOWN: Complete removal (cluster, config, everything)
task solo:teardown                       # Full cleanup

# Custom versions
task solo:start -- --consensus-node-version v0.70.0
task solo:start -- --mirror-node-version v0.146.0
task solo:start -- --consensus-node-version v0.70.0 --mirror-node-version v0.146.0

# Use local build (overrides consensus-node-version)
task solo:start -- --local-build-path ../hiero-consensus-node/hedera-node/data

# Check if Solo is running
task solo:status

# View logs
task solo:logs

# Run all integration tests
task test:integration

# Run specific test types
task test:integration:node
task test:integration:browser

# Stop Solo services
task solo:stop
```

## Default Versions

-   **Consensus Node:** v0.69.1 (or use `--local-build-path` for local builds)
-   **Mirror Node:** v0.145.2
-   **Default Nodes:** 1 node (use `task solo:init -- --num-nodes 2` for 2 nodes)

Override with `--consensus-node-version` and/or `--mirror-node-version` flags on `task solo:start`.  
Use `--local-build-path` for testing with a local Hiero consensus node build.

## Environment Variables

Located in `.env` (auto-generated by `task solo:start`):

```bash
HEDERA_NETWORK=local-node
OPERATOR_ID=0.0.XXXX          # Your test account
OPERATOR_KEY=302e...          # ECDSA private key
GENESIS_OPERATOR_ID=0.0.2     # Genesis account
GENESIS_OPERATOR_KEY=302e...  # Genesis key
```

## Common Issues & Quick Fixes

### Docker not running

```bash
# Start Docker Desktop, then:
task solo:init
task solo:start
```

### Port conflicts

```bash
pkill -f "kubectl port-forward"
task solo:start
```

### Tests failing

```bash
# Check Solo is running:
task solo:status

# Check .env exists:
cat .env

# Fresh start:
task solo:stop
task solo:start

# OR complete reset:
task solo:teardown
task solo:init
task solo:start
```

### Pods not starting

```bash
# Check pod status:
kubectl get pods -n solo

# View pod logs:
kubectl logs -n solo <pod-name>

# Full cleanup:
task solo:teardown
kind delete cluster --name solo-cluster
rm -rf ~/.solo
task solo:init
task solo:start
```

## First Time Setup Workflow

### Single Node Setup (Default, 12 GB RAM)

```bash
# 1. Install project dependencies (REQUIRED FIRST!)
task install

# 2. Initialize Solo infrastructure (one-time, ~1 minute)
task solo:init

# 3. Start Solo services (~8 minutes)
task solo:start

# 4. Verify setup
task solo:status
cat .env

# 5. Run tests
task test:integration

# 6. Stop when finished (or teardown for complete removal)
task solo:stop
```

### Dual Node Setup (For DAB Tests, 24 GB RAM)

```bash
# 1. Install project dependencies (REQUIRED FIRST!)
task install

# 2. Initialize Solo with dual nodes (one-time, ~2 minutes)
task solo:init -- --num-nodes 2

# 3. Start Solo services (~30 seconds)
task solo:start

# 4. Configure hosts for dynamic address book tests (REQUIRED for DAB tests)
echo "127.0.0.1 network-node1-svc.solo.svc.cluster.local" | sudo tee -a /etc/hosts
echo "127.0.0.1 envoy-proxy-node1-svc.solo.svc.cluster.local" | sudo tee -a /etc/hosts
echo "127.0.0.1 network-node2-svc.solo.svc.cluster.local" | sudo tee -a /etc/hosts
echo "127.0.0.1 envoy-proxy-node2-svc.solo.svc.cluster.local" | sudo tee -a /etc/hosts

# 5. Verify setup
task solo:status
cat .env

# 6. Run tests
task test:integration

# 7. Stop when finished (or teardown for complete removal)
task solo:stop
```

**Notes:**

-   You must run `task install` before setup to install Solo as a dependency.
-   The `/etc/hosts` configuration is only required for dynamic address book tests with dual-node setup.
-   `task solo:init` is a ONE-TIME operation. After that, use `task solo:start` and `task solo:stop` for daily work.

## Daily Development Workflow

```bash
# Morning - Start services (~30 seconds)
task solo:start

# During development
task test:integration:node     # Run Node.js tests
task test:integration:browser  # Run browser tests

# View logs if needed
task solo:logs

# End of day - Stop services (~5 seconds)
task solo:stop
```

## Kubernetes Commands (Advanced)

```bash

kubectl get pods -n solo

# List all services
kubectl get services -n solo

# Describe a pod
kubectl describe pod <pod-name> -n solo

# View pod logs
kubectl logs -n solo <pod-name> -f

# Restart port forwarding manually
kubectl port-forward svc/haproxy-node1-svc -n solo 50211:50211
```

## Account Types

### Standard Test Account (ECDSA)

-   **Use for:** 99% of integration tests
-   **Variables:** `OPERATOR_ID`, `OPERATOR_KEY`
-   **Type:** ECDSA
-   **Balance:** 10M HBAR

### Genesis Account

-   **Use for:** Genesis-specific operations only
-   **Variables:** `GENESIS_OPERATOR_ID`, `GENESIS_OPERATOR_KEY`
-   **Type:** ED25519
-   **ID:** Always `0.0.2`
